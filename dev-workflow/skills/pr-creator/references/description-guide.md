# Description生成ガイドライン

## 情報源の優先度

1. `.claude/plans/{issueId}.md` - タスク詳細ファイル（最優先）
2. Linear Issue - 課題の説明・背景
3. コミット履歴・差分 - 実際の変更内容

## デフォルトフォーマット（テンプレートがない場合）

```markdown
## 概要
<!-- このPRで何を実現するか、1-2文で -->

## 変更内容の詳細
<!-- 技術的な説明、なぜこのアプローチを選んだか -->

## スクリーンショット
<!-- UI変更がある場合は添付。ない場合は「なし」 -->

## 備考
<!-- レビュアーへの補足、関連Issue/PRなど -->
```

## 各セクションのポイント

### 概要
- 何を変更したか（What）を1-2文で
- タスク詳細ファイルの「概要」を参考にする

### 変更内容の詳細
- 技術的な変更内容とアプローチの理由（Why）
- タスク詳細ファイルの「実装計画」「調査結果」を参考にする
- 影響範囲

### スクリーンショット
- UI変更あり: `<!-- スクリーンショットをここに貼ってください -->` と案内
- UI変更なし: 「なし」

### 備考
- Breaking changesがあれば明記
- 関連Issue番号（`Closes #123`等）
- 今後の対応予定
