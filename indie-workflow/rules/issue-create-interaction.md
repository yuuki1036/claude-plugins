# Issue 作成時の方針確認ルール

`indie-issue-create` スキル実行時、後から変えにくい判断を伴う場面では `AskUserQuestion` による選択 UI で確認する。

## 共通原則

- **1件ずつ提示**: 複数の確認事項を一度にまとめない
- **what/why を先に**: 選択肢より先に「何を聞いているか・なぜ確認が必要か」をテキストで明記する
- **Other（自由入力）を常に含める**: 選択肢が用意できる場面でも自由入力の逃げ道を用意する
- **preview の提供**: テンプレート選択のような判断には、選択肢ごとの簡単な用途説明を添える

## 対象となる確認

### テンプレート選択（Phase 4）

自動判定の確信度が低い場合に AskUserQuestion で確認する。

- **what**: このタスクに使用するテンプレートを確認する
- **why**: テンプレート選択で Issue ファイルの構造が変わる。一度作ると直すのが手間なため最初に確認する
- **preview**: 自動判定した type とその判断根拠を1行で示す
- **選択肢**:
  1. bugfix — バグ修正・typo・設定変更など影響範囲が限定的
  2. feature — 新機能追加・既存機能の改修・リファクタリング
  3. investigation — 原因調査・パフォーマンス分析・技術選定
  4. debt — コード品質改善・依存関係更新・非推奨 API の移行
  5. その他（自由入力）

### scope_size 選択（Phase 4: feature の場合のみ）

テンプレート選択で feature が選ばれた直後に AskUserQuestion で確認する。

- **what**: この feature タスクの実装規模を選択する
- **why**: scope_size は Issue ファイルの frontmatter に記録され、タスク数の上限と作業量の見積もり基準になる
- **選択肢**:
  1. small — 3タスク以下（1-2日で完了する粒度）
  2. medium — 7タスク以下（数日〜1週間の作業）
  3. large — 15タスク以下（1週間以上の大きな機能）
  4. その他（自由入力）

### feature-dev 連携（Phase 7）

Issue ファイル作成・ブランチ切り替え完了後に AskUserQuestion で確認する。

- **what**: feature-dev で実装計画を立てるか
- **why**: ブランチを切った直後が最も作業コンテキストがそろっている。後回しにすると計画を立て直す手間が増える
- **選択肢**:
  1. はい、feature-dev で実装計画を立てる
  2. いいえ、後で自分でやる
  3. その他（自由入力）
