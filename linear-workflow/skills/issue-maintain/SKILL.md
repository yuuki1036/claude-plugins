# Issue Maintain

セッションの作業内容を Issue ファイルに反映し、品質整理と knowledge 切り出しを行う。

## ワークフロー

### Phase 1: 対象特定

1. 引数で Issue ID が指定されていればその Issue ファイルを対象にする
2. 指定がなければ、git branch で現在のブランチから Issue ID を抽出（例: `feat/CFP-123` → `CFP-123`）
3. それも取れなければ、`issues/` 配下の `status: in-progress` な全ファイルを対象にする
4. プロジェクトスラッグを特定（`CFP-123` → `cfp`）し、`.claude/linear/{slug}/issues/` からファイルを読み込む

### Phase 2: セッション内容の反映

会話コンテキストから今のセッションで行った作業を把握し、Issue ファイルに反映する:

1. **進捗チェックリスト更新** - 完了したサブタスクにチェックを入れる、新たに判明したサブタスクを追加
2. **設計判断の記録** - セッション中に決めた方針・アプローチを「計画」セクションに追記
3. **調査結果の記録** - 発見した事実・根本原因などを適切なセクションに追記
4. **変更ファイルの追記** - 変更したファイルパスを記録（必要に応じて）
5. **更新履歴にセッションサマリーを追記** - 日付と作業内容の1行サマリー

### Phase 3: 品質分析・整理

[quality-checklist.md](references/quality-checklist.md) に基づいて品質問題を洗い出し、整理する:

1. **完了サブタスクの圧縮** - 全サブタスク完了済みの詳細セクションを1行サマリーに
2. **重複除去** - 情報の正本を1箇所に決め、他は削除
3. **不採用・解決済みの削除** - 不要になった記述を削除
4. **フロントマター修正** - 不足フィールドの追加、status の更新
5. **セクション構成の調整** - type（bugfix/feature/investigation）に応じた構成に整理

変更が多い場合はユーザーに承認を求めてから実行する。

### Phase 4: Knowledge 切り出し

汎用性のある知見がセッション中に得られた場合:

1. **正確性の検証** - コードベース（Grep/Read）や関連 Issue と照合して内容が正しいか確認
2. **既存 knowledge との重複チェック** - `knowledge/` 配下の既存ファイルと重複しないか確認
3. **ファイル作成** - 検証済みの知見を `.claude/linear/{slug}/knowledge/{topic}.md` に作成
4. **元 Issue からの参照** - 切り出した知見への参照を元 Issue に残す（`→ knowledge/{topic}.md に切り出し済み`）

### Phase 5: 完了報告

実施した変更のサマリーを報告:
- 反映したセッション内容（更新した進捗、追記した設計判断など）
- 圧縮・削除した記述の数
- 切り出した knowledge ファイル
- 残存する「要確認」「TODO」項目（あれば）
