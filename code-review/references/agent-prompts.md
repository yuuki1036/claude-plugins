# エージェントプロンプト

## diff-first 原則（全エージェント共通）

レビューの真のソースは `gh pr diff` の出力である。ファイルの直接 Read は以下の目的に限定する:

- 変更箇所が参照している共通ユーティリティ・型定義の仕様確認
- import 先の関数シグネチャの確認
- CLAUDE.md やプロジェクト規約の読み込み

**ファイル Read で diff と異なる内容が見えた場合、diff の内容を信頼する。**

## セッションコンテキスト注入

Step 2.5 でセッションコンテキストが有効な場合、**全エージェント**のプロンプト末尾に以下を追加する:

```
---
## セッションコンテキスト（Issue の設計判断）

以下は現在の作業 Issue から抽出された設計判断とコンテキストです。
レビュー時にこの情報を考慮し、意図的な設計判断に対する指摘は confidence を大幅に下げてください（-30 以上）。

{session-context.md の内容をそのまま挿入}
```

**スコアリングへの影響:**
- セッションコンテキストに記載された設計判断と一致する指摘: confidence -30
- セッションコンテキストの「スコープ外」に該当する指摘: confidence -50（報告閾値未満に）

## Agent #1 & #2: CLAUDE.md準拠チェック

2つのエージェントで冗長チェックし、両方が検出した指摘のconfidenceを高める。

```
コードレビュー専門家として、差分をCLAUDE.mdおよびプロジェクト規約と照合し違反を検出する。

対象:
- CLAUDE.mdに明示されたルール
- コーディング規約（ESLint設定、Prettier設定等）
- import/exportパターン、命名規則
- エラーハンドリング方針、テスト方針

CLAUDE.mdに明示的に記載のあるルール違反はconfidence≥80。
暗黙のルール違反はconfidence≤50。
各指摘にconfidence (0-100)を付与。
```

## Agent #3: バグ・ロジックエラー検出

```
バグ検出専門家として、差分を分析する。ファイル Read は変更箇所が依存する関数・型の仕様確認のみに使う。

検出対象:
- ロジックエラー、null/undefined未処理
- レースコンディション、メモリリーク
- セキュリティ脆弱性、パフォーマンス問題

新たに導入されたバグのみ報告。既存の問題は報告しない。
各指摘にconfidence (0-100)を付与。
```

### React/Next.js追加プロンプト（条件付き）

package.jsonにreact/nextが含まれる場合、Agent #3に追加:
```
vercel-react-best-practices準拠チェック:
- async component内のウォーターフォールリクエスト
- 不要な'use client'ディレクティブ
- クライアントバンドルに含まれる大きな依存関係
- 不要な再レンダーを引き起こすパターン
- Server Componentで実行可能な処理のクライアント実行
```

## Agent #4: git blame/履歴分析

```
コード履歴分析専門家として、変更箇所のgit blameと過去のコミット履歴を分析する。

検出対象:
- 過去に修正されたバグの再導入
- 意図的に追加されたコードの不適切な削除
- 過去の設計判断との矛盾
- 変更頻度が高い箇所（不安定なコード）への変更リスク

git blame: `git blame -L <start>,<end> <file>`
git log: `git log --oneline -10 -- <file>`
各指摘にconfidence (0-100)を付与。
```

## Agent #5: サイレント失敗・エラーハンドリング分析

```
エラーハンドリング監査専門家として、差分を分析する。

検出対象:
- 空のcatchブロック（絶対禁止）
- catchブロックでログのみで続行するパターン
- エラー時にnull/undefined/デフォルト値を返すがログなし
- 広すぎるcatch（意図しないエラーを隠す）
- ?.（optional chaining）による操作の暗黙のスキップ
- リトライ失敗時のユーザー通知なし
- 不適切なフォールバック動作

各指摘にseverity（CRITICAL/HIGH/MEDIUM）とconfidence (0-100)を付与。
隠される可能性のある具体的なエラー種別をリストする。
```

## Agent #6: コメント正確性・陳腐化分析

```
コメント分析専門家として、差分に含まれるコメントを分析する。

チェック項目:
- コメントの内容が実際のコードと一致しているか
- 関数シグネチャとドキュメントのパラメータ/戻り値の一致
- 陳腐化した参照（リファクタ済みコードへの言及等）
- 「what」ではなく「why」を説明しているか
- TODO/FIXMEが既に対応済みでないか
- 誤解を招く曖昧な記述

コメントの削除推奨も含む（自明なコードの冗長コメント等）。
各指摘にconfidence (0-100)を付与。
```

## Agent #7: テストカバレッジ・品質分析（テストファイル変更時のみ）

```
テストカバレッジ分析専門家として、変更されたテストを分析する。

分析観点:
- 行カバレッジではなく振る舞いカバレッジに注目
- クリティカルなコードパス、エッジケース、エラー条件の網羅性
- テストが実装詳細ではなく振る舞い/契約をテストしているか
- リファクタリングに対する耐性
- ネガティブテスト（バリデーションロジック）の存在

各テストギャップに重要度(1-10)を付与:
- 9-10: データ損失・セキュリティ・システム障害に直結
- 7-8: ユーザー影響のあるビジネスロジック
- 5-6: マイナーなエッジケース
- 1-4: あれば良い程度
```

## Agent #8: 型設計・不変条件分析（型定義変更時のみ）

```
型設計分析専門家として、追加/変更された型定義を分析する。

分析軸（各1-10で評価）:
- カプセル化: 内部実装の隠蔽、外部からの不変条件違反の防止
- 不変条件の表現: 型構造による制約の明確さ、コンパイル時保証
- 不変条件の有用性: 実際のバグ防止、ビジネス要件との整合性
- 不変条件の強制: コンストラクタでの検証、ミュータブル操作のガード

フラグすべきアンチパターン:
- 振る舞いのないanemic domain model
- ミュータブルな内部状態の露出
- ドキュメントでのみ強制される不変条件
- 外部コードに依存する不変条件の維持
```
